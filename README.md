# Исследование сортировок строк

Практическое исследование обычных и строковых алгоритмов сортировки.  
Сравниваются время работы и количество посимвольных сравнений на массивах строк.

## Что реализовано

- обычный QuickSort;
- обычный MergeSort;
- тернарный String QuickSort;
- String MergeSort с LCP;
- MSD Radix Sort;
- MSD Radix Sort с переключением на String QuickSort.

Генератор строк использует алфавит из латинских букв, цифр и символов `!@#%:;^&*()-.`.
Длины строк: от 10 до 200 символов. Размеры массивов: от 100 до 3000 с шагом 100.

Проверяются три типа данных:

- случайные строки;
- обратно отсортированные строки;
- почти отсортированные строки.

## Структура

- `src/StringGenerator.*` — генерация тестовых массивов;
- `src/StringSortTester.*` — замер времени и посимвольных сравнений;
- `src/sorts.*` — реализации сортировок;
- `main.cpp` — запуск эксперимента и запись CSV;
- `results/measurements.csv` — результаты замеров;
- `plots/plot.py` — построение графиков;
- `plots/*.png` — готовые графики;
- `codeforces/*.cpp` — отдельные решения для задач A1m, A1q, A1r, A1rq.

## Как проводились замеры

Для каждого алгоритма и размера массива выполняется несколько запусков.  
В CSV записывается медианное время по 11 запускам, чтобы случайные выбросы меньше портили графики.

Количество посимвольных сравнений считается вручную внутри реализаций сортировок.
Для radix-сортировки это фактически количество обращений к символам.

## Краткий вывод

На случайных и почти отсортированных данных String QuickSort обычно показывает лучшее время.
MSD Radix Sort делает мало обращений к символам, но из-за накладных расходов не всегда быстрее по времени.
String MergeSort с LCP уменьшает число сравнений, но проигрывает по времени из-за копирования строк и дополнительных массивов.
